# NautiCAI Agentic AI – Integration Guide

This document explains what the **NautiCAI_AgenticAI** folder contains (and what it does **not**), how it differs from the current NautiCAI backend, and how to integrate it so the web app uses the agentic pipeline and its PDFs.

---

## 1. What’s actually in the folder (and what’s not)

### Implemented in the code you have

| Feature | Where | What it does |
|--------|--------|----------------|
| **Hull inspection (YOLO + SAM)** | `nauticai_hull_inspection.py` | Detects biofouling with YOLO, segments with SAM → coverage %, severity, per-detection data (masks, crops, annotated image). |
| **REST API** | `nauticai_api.py` | FastAPI server: upload image + vessel_id → run pipeline → return JSON. |
| **IMO compliance logic** | `nauticai_api.py` | Maps coverage % to IMO rating (FR-0/FR-2/FR-3/FR-4), recommended action, requires_cleaning. |
| **Server-generated PDF report** | `nauticai_api.py` | FPDF “Official Hull Inspection Report”: vessel details, hull condition, compliance rating, required action. Saved as `{vessel_id}_Audit_Report.pdf`. |
| **Report storage** | `nauticai_api.py` | File-based: `reports/{vessel_id}_inspection_data.json` and `reports/{vessel_id}_Audit_Report.pdf`. |
| **Endpoints** | `nauticai_api.py` | `POST /api/inspect`, `GET /api/vessel/{id}/latest-report`, `GET /api/vessel/{id}/pdf`, `GET /api/vessels/all`, `/`, `/health`, `/docs`. |

So in the code you have: **inspection pipeline + IMO logic + PDF generation + REST API**. No other automation is implemented in these files.

### In requirements but not used in the code

| Dependency | In code? | Note |
|------------|----------|------|
| **python-telegram-bot** | **No** | Listed in `requirements.txt` under “Telegram Bot”, but **nothing in nauticai_api.py or nauticai_hull_inspection.py imports or uses it**. There is no Telegram bot, no “send automated Telegram message” logic in this folder. |
| **nest-asyncio** | **No** | Also in requirements; not referenced in the provided code. Often used with bots/async loops. |

So: **automated Telegram messages (or any Telegram integration) are not in this codebase.** They may exist in another repo/branch, or were planned and never added to this zip. Worth confirming with Charan: “Is there a separate script or repo for the Telegram bot / automated messages?”

## 2. What’s in the AgenticAI folder (file-by-file)

**Location:** `NautiCAI_AgenticAI/NautiCAI AI/NautiCAI_Backend/`

| File / folder | Role |
|---------------|------|
| **README.md** | Setup: add `biofouling_best.pt` and `sam_vit_b_01ec64.pth` (in `sam_checkpoints/`), `pip install -r requirements.txt`, `python nauticai_api.py` → server on port 8000. |
| **nauticai_api.py** | **FastAPI server** for the agentic backend. Loads YOLO + SAM on startup, exposes REST API. **Creates IMO-style PDF reports** (FPDF) and stores JSON + PDF under a `reports/` folder keyed by `vessel_id`. |
| **nauticai_hull_inspection.py** | **AI vision pipeline**: YOLO (biofouling detection) → **SAM (Segment Anything)** for segmentation → coverage %, severity, per-detection data. Writes annotated images, masks, crops, and per-image JSON. |
| **requirements.txt** | Dependencies: FastAPI, uvicorn, torch, ultralytics, opencv, segment-anything (Git), **fpdf** (PDF). Optional: python-telegram-bot. |
| **sam_checkpoints/** | Place **sam_vit_b_01ec64.pth** here. |
| **test_images/** | Sample image for testing. |

**Important:** The agentic backend is **vessel-centric** and **file-based**: one `vessel_id` per inspection, reports stored as files (`reports/{vessel_id}_inspection_data.json`, `reports/{vessel_id}_Audit_Report.pdf`). It does **not** use Postgres or your current auth.

## 3. Agentic API vs current app API

### Agentic backend (Charan’s)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/` | GET | Service info |
| `/health` | GET | Health check |
| **`/api/inspect`** | **POST** | **Upload hull image + vessel_id (form).** Runs YOLO+SAM pipeline, applies IMO logic, saves JSON + **generates PDF**, returns JSON report. |
| `/api/vessel/{vessel_id}/latest-report` | GET | Latest inspection JSON for vessel |
| **`/api/vessel/{vessel_id}/pdf`** | **GET** | **Download the PDF** generated by the agentic pipeline |
| `/api/vessels/all` | GET | List all vessels (from JSON files in `reports/`) |
| `/docs` | GET | Swagger UI |

- **PDF:** Created **on the server** with FPDF (IMO-style: vessel details, hull condition, compliance rating, required action). Stored as `{vessel_id}_Audit_Report.pdf`.
- **Inspect input:** `vessel_id` (form) + `image` (file). No auth in the provided code.

### Current NautiCAI app

| Endpoint | Method | Purpose |
|----------|--------|---------|
| **`/detect`** | **POST** | Upload **file** (+ optional vessel_name). Returns **DetectResponse** (inspection_id, detections, annotated_image base64, summary). Saves to **Postgres** (inspections table). |
| **`/inspections`** | GET | List inspections from **Postgres** (optionally by inspection_id). |
| **`/inspections/{id}`** | DELETE | Delete inspection by DB id. |
| `/health` | GET | Health + DB check. |

- **PDF:** Built **in the frontend** (jsPDF) from inspection data + optional annotated image. No server-side PDF.
- **Data model:** **inspection_id** (e.g. NCR-20260315-xxxx), DB primary key `id`, file_name, risk_level, etc. Stored in Postgres.

So:

- **Agentic** = vessel_id, file storage, **server-generated IMO PDF**.
- **Current** = inspection_id, Postgres, **client-generated simple PDF**.

## 4. Role of each part for integration

- **nauticai_hull_inspection.py**  
  - **Role:** Run the “agentic” inspection: YOLO + SAM → coverage, severity, detections.  
  - **Use:** Either call this from your **existing** backend (api.py) or run the agentic server and have the app call it.

- **nauticai_api.py**  
  - **Role:** Exposes `/api/inspect` (run pipeline + IMO logic + save JSON + **create PDF**) and `/api/vessel/{vessel_id}/pdf` (download that PDF).  
  - **Use:** So that “PDFs created by agentic” = the FPDF report from this server. The web app should “download PDF” from here when using the agentic flow.

- **PDF (create_pdf in nauticai_api.py)**  
  - **Role:** Official IMO-style report (vessel details, hull condition, compliance rating, required action).  
  - **Use:** This is the PDF you want users to download when you “integrate agentic”; the current frontend jsPDF report is the non-agentic version.

## 5. How to integrate with the web application

High-level: **either** run the agentic backend alongside the current one and point the app at it for “agentic” inspections and PDFs, **or** merge the agentic pipeline (and optional PDF) into the current backend. Below is the clearest path so that “agentic creates the PDF” and “that PDF is what we download.”

### Option A – Two backends (fastest to “agentic PDF in app”)

1. **Run both servers**  
   - Current app backend: e.g. port **8000** (or keep as now).  
   - Agentic backend: port **8001** (e.g. `python nauticai_api.py` with `uvicorn ... --port 8001`).

2. **Add model files to Agentic folder**  
   - `biofouling_best.pt` in `NautiCAI_AgenticAI/NautiCAI AI/NautiCAI_Backend/`.  
   - `sam_vit_b_01ec64.pth` in `.../NautiCAI_Backend/sam_checkpoints/`.  
   - Install: `pip install -r requirements.txt` in that backend folder.

3. **Frontend: “Use Agentic” flow**  
   - **Inspect (run inspection):**  
     - When user runs an inspection and you want agentic:  
       - POST to **agentic** `http://<agentic-host>:8001/api/inspect` with `vessel_id` + `image`.  
       - Use a `vessel_id` that you can later use for PDF (e.g. inspection_id or a new vessel id you store).  
     - Store in your app state (or minimal backend) that “this inspection’s PDF is from agentic” and the `vessel_id` used for that request.  
   - **Download PDF:**  
     - If “this inspection used agentic”:  
       - **GET** `http://<agentic-host>:8001/api/vessel/{vessel_id}/pdf` → that’s the **agentic-generated PDF**; serve or redirect the user to download it.  
     - Else: keep current behavior (frontend builds PDF with jsPDF from inspection data).

4. **Dashboard / Reports**  
   - Can stay on current backend (Postgres) for list/detail/delete.  
   - For rows that came from agentic, store `vessel_id` (or a flag) so the “Download PDF” button calls the agentic PDF endpoint instead of building jsPDF.

So: **agentic creates the PDF on its server; the web app just downloads it from `/api/vessel/{vessel_id}/pdf`.**

### Option B – Single backend (merge into api.py)

1. **Copy agentic pipeline into current backend**  
   - Copy `nauticai_hull_inspection.py` into your backend (e.g. next to `api.py`).  
   - In `api.py` (or a separate router), on a new endpoint (e.g. `POST /api/agentic/inspect`):  
     - Accept vessel_id + image.  
     - Call `vision_pipeline.load_models(...)` once at startup (reuse existing YOLO/SAM paths or env).  
     - Call `vision_pipeline.process_image(temp_path)`.  
     - Apply IMO logic (copy from nauticai_api.py).  
     - Save JSON under a `reports/` (or configurable) folder.  
     - Call `create_pdf(...)` (copy the FPDF class and `create_pdf` from nauticai_api.py) so the **same** IMO PDF is generated.  
   - Add `GET /api/vessel/{vessel_id}/pdf` that returns the file from disk (same as agentic).

2. **Dependencies**  
   - Add to current backend: `fpdf`, and SAM dependency (segment-anything).  
   - Ensure `biofouling_best.pt` and `sam_vit_b_01ec64.pth` are in the right paths (or env).

3. **Frontend**  
   - Same as Option A for “run agentic inspection” and “download PDF”: call the new `/api/agentic/inspect` and `/api/vessel/{vessel_id}/pdf` (or keep path under `/api/...`).

4. **Unify with Postgres (optional)**  
   - After running the agentic pipeline, you can also insert a row into your existing `inspections` table (e.g. inspection_id = vessel_id or a new id) so Dashboard/Reports still list it; “View PDF” then uses the server-generated PDF URL for that vessel_id.

## 5. Data mapping (current app ↔ agentic)

- **Current:** `inspection_id` (e.g. NCR-…), `id` (DB), `file_name`, `risk_level`, `detected_classes`, `created_at`.  
- **Agentic:** `vessel_id`, `reports/{vessel_id}_inspection_data.json`, `reports/{vessel_id}_Audit_Report.pdf`, and JSON with `metadata`, `ai_vision_metrics`, `compliance_result`.  

To avoid confusion:

- Decide one **id** to use as `vessel_id` when calling agentic (e.g. same as `inspection_id`, or a new “vessel” id per inspection).  
- When listing inspections in the app, for “agentic” rows you can store `vessel_id` (or a flag) and use that in the “Download PDF” link to hit `/api/vessel/{vessel_id}/pdf`.

---

## 6. Summary

- **What’s in the folder:** Standalone FastAPI server + YOLO+SAM vision pipeline + IMO logic + **server-side FPDF report** and file-based storage.  
- **Role:** Run inspections with segmentation/coverage and **generate the official PDF on the server**; expose it via `/api/vessel/{vessel_id}/pdf`.  
- **Integration:** Use **Option A** (two backends) to quickly wire the app so “agentic creates the PDF” and “download” uses that PDF; or **Option B** (merge into api.py) for one server and one deployment. In both cases, “how PDFs are created” for the agentic flow is: **agentic backend (or merged code) creates them; the web app only downloads them.**

Once you choose Option A or B, the next steps are: add the two model files, install agentic requirements, then implement the frontend calls (inspect → agentic, download → agentic PDF) and optionally link agentic results to Dashboard/Reports via vessel_id or inspection_id.
